# 分支条件追踪参考文档

本文档说明分支条件追踪的**核心原则**。LLM 应根据代码语义判定分支影响，而非匹配固定模式。

---

## 1. 核心判定原则

### 1.1 分支追踪的目的

确定在什么条件下敏感操作会/不会执行，避免漏洞误判。

### 1.2 判定思路

```
1. 识别代码中的条件分支（if/else/switch/三元表达式）
2. 检查每个分支的执行内容
3. 判定敏感操作在哪些分支中执行
4. 输出执行路径结论
```

### 1.3 关键判定项

| 判定项 | 影响 |
|:-----|:-----|
| 敏感操作在哪个分支？ | 决定是否需要满足特定条件 |
| 条件是否可控？ | 决定攻击者能否选择执行路径 |
| 是否有提前退出？ | 可能阻止代码执行到敏感操作 |

---

## 2. 必须识别的分支类型

**根据代码语义识别，而非匹配固定方法名：**

| 分支类型 | 语义特征 | 追踪要点 |
|:---------|:---------|:---------|
| **环境/平台分支** | 代码根据运行环境选择不同执行路径 | 标注各环境下的执行内容 |
| **安全检查分支** | 代码对输入进行安全验证后决定是否继续 | 标注拦截条件和绕过可能 |
| **功能开关分支** | 代码根据配置/开关决定是否执行功能 | 标注开关状态的影响 |
| **空值/异常分支** | 代码检查参数有效性后决定是否继续 | 标注提前退出条件 |
| **权限判断分支** | 代码检查用户权限后决定是否允许操作 | 标注权限要求 |
| **协议/格式分支** | 代码根据输入格式选择不同处理方式 | 标注格式限制 |

---

## 3. 提前退出点识别

### 3.1 退出类型

| 退出类型 | 语义 | 影响 |
|:---------|:-----|:-----|
| **安全拦截退出** | 安全检查失败，拒绝继续 | 阻止敏感操作执行 |
| **空值保护退出** | 参数无效，提前返回 | 敏感操作不执行 |
| **错误抛出退出** | 条件不满足，抛出错误 | 流程中断 |
| **条件返回退出** | 特定条件下返回，不执行后续代码 | 敏感操作可能不执行 |

### 3.2 识别要点

检查代码时，关注：
- `return` 语句的位置和条件
- `throw` 语句的位置和条件
- 条件分支中哪些路径会到达敏感操作

---

## 4. 输出格式

### 4.1 分支结构标注

```markdown
### [Level N] ClassName.methodName()

**分支结构:**
```
methodName(param)
    ├─ [if 条件A] → 分支A执行内容
    │      └─→ 敏感操作 ⚠️
    ├─ [else if 条件B] → 分支B执行内容
    └─ [else] → 默认分支 / return / throw
```

**分支判定:**
| 分支 | 条件 | 是否执行敏感操作 |
|:-----|:-----|:-----------------|
| 分支A | 条件A为真 | ✅ 执行 |
| 分支B | 条件B为真 | ❌ 不执行 |
```

### 4.2 执行路径结论

```markdown
## 执行路径结论

| 路径 | 条件 | 敏感操作 | 可利用性 |
|:-----|:-----|:---------|:---------|
| 路径A | {条件} | ✅ 执行 | 可利用 |
| 路径B | {条件} | ❌ 不执行 | 不可利用 |

**审计提示:**
- 环境确认: {需确认的环境条件}
- 安全拦截: {识别到的检查点}
- 可利用条件: {满足什么条件时可利用}
```

---

## 5. 判定原则总结

| 原则 | 说明 |
|:-----|:-----|
| **语义优先** | 识别代码意图，不匹配固定模式 |
| **路径完整** | 检查所有可能的执行路径 |
| **退出标注** | 明确标注所有提前退出点 |
| **条件明确** | 清晰说明每个分支的触发条件 |
| **结论清晰** | 明确输出哪些条件下敏感操作会执行 |
